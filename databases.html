Databases – SQLite Ticket Tracker

This page includes my narrative reflection and access to the original and enhanced versions of my SQLite Ticket Tracker artifact.

Project Overview

This artifact is a small support ticket tracking system built with Python and SQLite. The original version stored tickets in a basic table and supported simple create/read operations.

For the enhancement, I focused on making the database layer more reliable and “real-world” by improving data integrity, query safety, and performance. I updated the schema with constraints, added indexing for faster lookups, and used parameterized queries to reduce the risk of SQL injection. I also added reporting queries so the system can generate meaningful summaries instead of only storing data.

Narrative Reflection

Brief Description

This artifact was created as a ticket tracking system to store, update, and retrieve support tickets using SQLite. It demonstrates database design fundamentals, query construction, and handling structured data in an application.

Justification for Inclusion

I included this artifact because it shows practical database skills that translate directly into real software development. The enhanced version demonstrates stronger database design through constraints, indexing, and cleaner query structure. These improvements make the system more reliable, more secure, and easier to maintain.

Enhancements and Skills Demonstrated

  • Improved schema design: added constraints to protect data integrity and prevent invalid records.
  • Performance: added indexes to improve query speed for common filters and lookups.
  • Security mindset: used parameterized queries to reduce SQL injection risk.
  • Reporting/analytics: added queries that summarize ticket volume, categories, and resolution trends.
  • Code quality: organized database functions for clearer reuse and better maintainability.

Reflection on the Process

The biggest challenge was balancing simplicity with realism. SQLite is lightweight, but database design decisions still matter. By tightening the schema, adding indexes, and using safer query patterns, I learned how small improvements can make a system significantly more dependable. This also helped me think more like a developer who expects real user input and long-term maintenance, not just a “working demo.”
